    Executive summary

    The Google Drive folder contains four authoritative capstone documents (pre-proposal, approved scope/modules, progress report, security/project rubric). Together, they define iTECHS as a gamified e-learning platform for ICT students at Pasay City South High School, emphasizing secure authentication + role-based access control, admin-managed user records, game-based assessment, progress tracking, and AI-assisted hints that guide without revealing final answers. The rubric additionally requires secure CRUD, comprehensive validation, OTP verification, search/filter, and logging/monitoring of critical actions.

    Your updated MultimodalText requirements refine the product into a landscape-first PWA with a pixel-art “game menu” UI, Admin-created accounts only, Teacher oversight (review/configure difficulty + star settings, not edit questions), a fixed 20-level progression per student, 3 expandable mini-games, AI-generated question packs pre-generated for each new student for all levels, and a stars-based per-level mastery system plus a separate correctness-based overall score. Cosmetics are profile-only and purchased with “star currency.”

    For delivery speed with GitHub Copilot, the highest-leverage practice is to treat the UI as a small design system: define Figma components + variants + component properties, build with Auto Layout for responsive landscape panels, and map design tokens to code using Figma Variables + code syntax surfaced in Dev Mode. Figma explicitly positions Auto Layout as making designs “responsive and adaptable,” and Dev Mode as the developer handoff surface providing code snippets and variable-aware inspection. [1] If your Figma plan supports it, Code Connect can connect real React components to design components so Dev Mode shows connected snippets; Figma describes Code Connect as a bridge between codebase components and Dev Mode, improving implementation specificity and AI handoff via the MCP server. [2]

    Figma research for pixel UI workflows and design-to-code mapping

    
    Using Auto Layout to make landscape UI reliably responsive

    Figma’s Auto Layout is designed to make frames respond dynamically to content changes, “saving you time and making your designs responsive and adaptable.” [3]For a landscape-first PWA, this supports a consistent structure:

    Outer “world background” frame: fixed background image; content overlays centered.

    Inner “panel” frames: Auto Layout for header row (Back/Title/Search), body region, and footer controls.

    Reusable “icon tile” buttons: Auto Layout horizontal/vertical to lock padding and label spacing consistently.

    Recommended Figma practice for this project: - Every panel (forms, tables, level select) should be a component built on Auto Layout; instance text should be a text component property so content replaces cleanly without resizing artifacts.

    
    
    Dev Mode and variable-aware handoff for Tailwind alignment

    Figma Dev Mode provides code snippets in the Inspect panel and includes layout/typography details; it also surfaces variables assigned to elements inside code snippets. [4]For variable-heavy designs, Figma documents that Dev Mode includes features like Variable details (including mode/value/aliases and code snippet for using the variable). [5]

    This matters because pixel UI needs consistent tokens for: - border thicknesses - panel padding - type sizes - shadows/outlines - star/currency colors

    Figma supports code syntax for variables, intended to represent variables in code using valid names and appear in Dev Mode code snippets for supported contexts (CSS is relevant for Tailwind projects). [6]Figma also documents importing design tokens into variables using a JSON format (DTCG), which is relevant if you want to keep Tailwind theme tokens and Figma tokens aligned via a shared token source file. [7]

    
    
    Mapping Figma components to React components with variants and component properties

    Figma’s “variants” are designed to define component attributes like size, state, and color as “variant properties and values,” and Figma notes you can map properties/values to code components in your design system. [8]Figma’s component properties support instance-level edits via typed property controls (boolean visibility toggles, instance swaps, text props, variant selection). [9]

    Best practice for your Copilot-ready workflow: - Build a Pixel UI component library in Figma using: - PixelButton variants: kind (primary/secondary/danger), state (default/hover/disabled), size (sm/md/lg) - PixelPanel variants: size and header on/off - IconTileButton variants: locked state and badge on/off - StarMeter variants: 0–3 filled - Use “text properties” for labels so designers can edit text without detaching components. [9]

    
    
    Code Connect for higher-fidelity Figma-to-React mapping

    If your plan allows it, Figma positions Code Connect as a bridge between your codebase components and Dev Mode, connecting repository components directly to design components, enhancing implementation detail and AI tooling context through the MCP server. [2]Figma also notes that Dev Mode’s default code snippets can be customized so developers see connected design system code instead of auto-generated code when inspecting components. [10]

    For your GitHub Copilot usage, this can materially reduce ambiguity by ensuring: - Figma components expose real prop names (React) and variant mappings - Copilot prompts can reference component contracts rather than re-inventing them

    High-level system flow

    flowchart TD  A[Admin creates Teacher/Student accounts] --> B[System sends OTP / activation]  B --> C{OTP verified within validity window?}  C -- No --> C1[Activation fails / retry OTP]  C -- Yes --> D[Account activated]  D --> E[If Student: provision profile + wallet + 20 levels]  E --> F[AI job generates QuestionPack for Levels 1-20]  F --> G{All packs READY?}  G -- No --> G1[Mark failed levels; allow Admin regen]  G -- Yes --> H[Student can play Level 1]  H --> I[Student starts Level session]  I --> J[Load assigned GameMode + QuestionPack]  J --> K[In-game: answer questions; mistakes + hints tracked]  K --> L[Compute stars + score for attempt]  L --> M[Persist GameSession + update StudentLevelState]  M --> N[Unlock next level if criteria met]  M --> O[Update Teacher/Admin dashboards + leaderboards]

    
    
    Level → game assignment logic

    flowchart TD  A[Provision new Student] --> B[Define LevelPairs: (1-2),(3-4)...(19-20)]  B --> C[Initialize previousGame = NONE]  C --> D{For each LevelPair}  D --> E[Candidate games = {A,B,C} minus previousGame]  E --> F[Select game from candidates using balanced random (shuffle bag)]  F --> G[Assign game to both levels in the pair]  G --> H[previousGame = selected game]  H --> D  D --> I[Persist LevelGameAssignment(studentId, levelNumber, gameMode)]
